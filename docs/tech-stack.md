# 🛠️ 技術スタックと選定理由

本プロジェクト（GardenTree Portfolio）の技術スタックと、なぜそれらを選定したのかの背景を開発者向けにまとめています。

## 基本コンポーネント

- **言語**: [TypeScript](https://www.typescriptlang.org/)
- **フレームワーク**: [Next.js (App Router)](https://nextjs.org/) ※現在 v16 / React v19 環境
- **UIライブラリ**: [React](https://react.dev/)
- **スタイリング**: [Tailwind CSS v4](https://tailwindcss.com/)
- **デプロイ環境**: [Vercel](https://vercel.com) (想定)

---

## 💡 選定理由詳細

### 1. Next.js (App Router)

ポートフォリオサイトは、一般的に「高速なページロード」「優れたSEO」「リッチなUI表示」が求められます。
Next.js は以下の点でポートフォリオ開発に最も適しています。

- **SSG / 自動最適化**: 外部APIや動的な頻繁なデータベースフェッチが不要な本サイトでは、ビルド時に静的HTMLとしてページを生成し、VercelのEdge CDNから高速に配信が可能です。
- **直感的なルーティング**: `src/app/` ディレクトリ配下に `page.tsx` を置くだけでルーティングされるためファイルの役割が明確です。また `projects/[slug]/page.tsx` のような動的ルートもシンプルに作成できます。
- **`next/image` による画像最適化**: サイトをリッチに見せるためにプロジェクト画像を多数配置しますが、それらの WebP 自動変換、次世代フォーマット配信、デバイスごとのリサイズなどを自動で行い、LCPの改善とCore Web Vitalsの最適化に大きく貢献します。

### 2. TypeScript

「個人開発であっても、時間を空けて触ったときに壊れない仕組み」を作るためです。

- **安全なデータバインディング**: プロフィールやプロジェクト等のテキストや画像URLは `src/data/` 以下のJSONファイルで一元管理しています。ここで `Project` や `Profile` のインターフェース型を厳密に定義することで、「プロパティのタイポ」や「未定義プロパティへのアクセス」などを書いている最中に検知でき、思わぬバグの発生を未然に防ぎます。
- **Vercelビルドでの水際防御**: デプロイ時に厳格な型チェック（`tsc`）が走るため、軽微なミスが本番環境へ流出するのを防げるというメリットは計り知れません。

### 3. Tailwind CSS (v4)

コーディングの「速度（DX）」と「デザインの統一感」を両立するためです。

- **ユーティリティファースト**: 別途CSSファイルを管理する必要がなく、JSXの構成とスタイリングを同時に記述できます。ファイル間の行き来が減り、思考を妨げずにレイアウトが組めます。
- **軽量でモダンなv4**: 最新の Tailwind CSS v4 では `tailwind.config.js` に依存せず CSS 単体で設定や変数が完結するモダンな構成となっており、プロジェクトをシンプルに保てます。
- **レスポンシブ・ダークモードの容易さ**: `sm:` や `lg:` によるモバイル/PC調整、`dark:` によるダークモード対応がクラス名をつけるだけで完結します。特にダークモード対応においては、システムへの追従も簡単なため、モダンなポートフォリオに欠かせない要件を手軽に満たせます。

### 4. Vercel

Next.jsの開発チームが提供している基盤であり、相性が最も良いためです。
GitHubのリポジトリを繋ぐだけで「プッシュするたびに自動で Lint/Type チェック → ビルド → デプロイ」という CI/CD サイクルが完成します。インフラの管理を気にせず、開発者はポートフォリオのコンテンツ拡充やブラッシュアップにのみ集中できます。
